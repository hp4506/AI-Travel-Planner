VagaWise — System Flow Description

Complete Phase-by-Phase Data Flow: User Input → Express Backend → Gemini AI → UI Render
Open in Chrome / Edge → Ctrl+P → Save as PDF

Overview

VagaWise is a full-stack web application where the user's input is collected on the browser frontend, serialised into a JSON object, and sent via HTTP POST to a Node.js/Express backend. The backend orchestrates calls to the Google Gemini AI (for reasoning), OpenWeatherMap API (for live weather), and ExchangeRate-API (for currency conversion). The combined response is returned as JSON and rendered dynamically in the browser.

The entire workflow is divided into four sequential phases, each triggered by a specific user action.

Phase 1 — User Input Collection

Frontend: app.js + index.html

The user fills in the trip planning form on index.html. JavaScript in app.js listens to all input events (destination selection, budget slider, date picker, preference tags) and assembles a single, structured JSON payload. Validation checks run before the payload is dispatched — ensuring destinations, budget, dates, and location are all present.

  • Budget Slider: A syncBudget() function keeps the hidden #budget input, the visual slider, and the manual text input all in sync. Any change in one propagates to the others instantly.
  • Multi-city Days: For multi-city trips, the user distributes total days per city. The total must sum exactly to the overall duration before proceeding.
  • Preferences: Tag buttons (Adventure, Food, Culture, etc.) toggle from the DOM and are collected as an array.
// JSON Payload assembled by app.js before Phase 2 begins: { "destinations": ["Goa", "Manali"], "budget": 70000, "days": 7, "currentLocation": "Hyderabad", "isInternational": false, "daysPerCity": [ { "city": "Goa", "days": 4 }, { "city": "Manali", "days": 3 } ], "preferences": ["adventure", "food"] }

Phase 2 — Budget Feasibility Analysis

Trigger: User clicks "Craft & Plan"

The Phase 1 JSON payload is dispatched via HTTP POST to /api/trips/analyze. The Express route handler in tripRoutes.js receives it and calls two services in sequence:

  • Step 1 – Gemini AI Call: geminiService.analyzeBudgetFeasibility() constructs a detailed natural-language prompt and sends it to the Gemini model. The prompt instructs the AI to behave as a travel cost analyst — estimating hotel, food, and transport costs for each city based on its real-world cost index, then determining if the user's budget is sufficient.
  • Step 2 – Currency Enrichment: For each city in the breakdown, currencyService.convertToLocalCurrency() calls the ExchangeRate-API to fetch the live INR conversion rate, adding a localCurrency object to each city's data.
  • Step 3 – Response: The enriched analysis JSON is returned to the frontend. renderAnalysis() in app.js displays the breakdown cards, feasibility status, and either the Transport Details form (if feasible) or clickable alternative destination cards (if not feasible).

Phase 3 — Itinerary Generation

Trigger: User fills Transport Details → clicks "Continue with AI Planning"

After budget approval, the user provides their travel logistics in a form: mode of transport (Flight / Train / Bus / Personal Car), departure time, and arrival time. App.js collects this along with the approved budget allocation and constructs a second, richer JSON payload. The payload is dispatched to /api/trips/plan.

  • Step 1 – Weather Fetch: weatherService.getDestinationsWeather() calls the OpenWeatherMap API for each city, retrieving the current temperature and weather condition.
  • Step 2 – Gemini AI Call: geminiService.generateItinerary() builds a comprehensive prompt embedding the transport context, nearest departure airport (auto-derived from currentLocation text by the AI), weather data, and per-city budget allocations. The AI generates a full multi-day, time-blocked schedule with specific hotel suggestions.
  • Step 3 – Auto-Save: Once the itinerary renders, the full trip object is automatically saved to Firebase Firestore under the logged-in user's ID via /api/trips/save.

Phase 4 — Dynamic Rescheduling (On Demand)

Trigger: User taps "Missed" on any activity card

Each activity rendered in the itinerary includes a "Missed" button. When tapped, markMissed(day, blockName, idx) in app.js fires. It extracts the missed activity object and the remaining days from currentTripData, then sends them to the backend for AI-powered re-balancing.

  • Step 1 – Gemini AI Call: geminiService.rescheduleItinerary() sends the missed activity, remaining schedule, and remaining budget to Gemini. The AI decides whether to shift the missed item to the next available slot (high priority) or skip it and replace it with an alternative, strictly maintaining the trip's end date and budget.
  • Step 2 – Merge & Re-render: The returned updated days are merged with the already-completed days (days before the missed one) and the full itinerary is re-rendered in the UI instantly.
